1. Architecture Overview

Pi is an extension-based agent system built on @mariozechner/pi-coding-agent. There is no single entry point — behavior is composed from extensions loaded via settings.json.

Core Concept: Extensions All the Way Down

Each .ts file in agent/extensions/ is an extension that receives an ExtensionAPI object. Extensions register:

  • Tools — callable by the LLM (e.g., run_chain, dispatch_agent)
  • Commands — user slash commands (e.g., /chain, /pipeline)
  • Shortcuts — keyboard bindings (e.g., Ctrl+B, Ctrl+G)
  • Event handlers — lifecycle hooks (session_start, before_agent_start, tool_call)
  • Widgets — live TUI components rendered in the terminal

The Primary Agent

The primary pi process is an orchestrator only. Its before_agent_start hook injects a system prompt that says:

"You are a dispatcher agent. You coordinate specialist agents..."
"You PREFER delegating codebase work to specialist agents via dispatch_agent"

The primary agent can use tools directly, but is prompted to delegate. Every delegation spawns a new pi subprocess.

2. The Three Pipeline Systems

Pi has three distinct orchestration modes. Only one is active at a time, selected via commands.

▶ Chain — Sequential Pipeline agent-chain.ts

Pattern: Step 1 → Step 2 → Step 3 → ... (output feeds forward as $INPUT)

Tool: run_chain — executes the entire chain sequentially

Config: .pi/agents/agent-chain.yaml defines chain names and step sequences

How it works:

  1. User's prompt enters Step 1 as $ORIGINAL
  2. Step 1's output becomes $INPUT for Step 2
  3. Each step spawns a fresh pi subprocess with the step's agent definition
  4. Agents maintain session context via JSON session files in .pi/agent-sessions/

Use case: Repeatable workflows like code audits (scout → analyze → report)

▶ Team — Parallel Dispatch agent-team.ts

Pattern: Primary dispatches specialist agents on demand (parallel or sequential)

Tool: dispatch_agent — sends one task to one specialist

Config: .pi/agents/teams.yaml defines team compositions; agent definitions in agents/*.md

How it works:

  1. Primary analyzes user's request and breaks it into sub-tasks
  2. Each sub-task is dispatched to the right specialist (scout, builder, reviewer, etc.)
  3. The LLM decides dispatch order — can run multiple agents concurrently
  4. Each dispatch spawns a new pi subprocess

Use case: Flexible task delegation where the LLM chooses the strategy

▶ Pipeline-Team — 5-Phase Hybrid pipeline-team.ts

Pattern: Fixed 5-phase pipeline with mixed interactive/parallel/sequential phases

Tools: advance_phase, dispatch_agents, pipeline_status

Config: .pi/agents/pipeline-team.yaml

Phases:

  1. UNDERSTAND (interactive) — Primary talks to user, clarifies task
  2. GATHER (parallel) — Scout agents explore the codebase concurrently
  3. PLAN (sequential) — Planner creates implementation plan from gathered context
  4. EXECUTE (parallel) — Builder agents implement the plan
  5. REVIEW (loop) — Reviewer audits, primary decides approve or re-dispatch

Use case: Complex features requiring structured multi-agent collaboration

4. Pipeline Deep Dive: The 5 Phases

When using pipeline-team.ts, every task walks through up to 5 sequential phases. Here's what each does and where time is spent:

UNDERSTANDinteractive
Primary agent converses with the user to clarify the task. No subprocesses spawned.
~5-30s (depends on back-and-forth)

The LLM must call advance_phase with a summary before moving on. Even for clear tasks, this involves at least one LLM round-trip to produce the summary.

GATHERparallel
Scout agents explore the codebase concurrently. Multiple pi subprocesses spawned.
~10-40s (2-5 scouts in parallel)
Bottleneck: Subprocess Startup

Each scout spawns a new pi process. Startup cost: ~2-3s per process (Node.js boot + extension loading + API handshake).

PLANsequential
Planner agent synthesizes gathered context into an implementation plan. Single subprocess.
~15-45s (one agent, potentially multiple LLM turns)

The $CONTEXT variable carries all accumulated context (capped at 15,000 chars). Planner's output is stored as $PLAN for the execute phase.

EXECUTEparallel
Builder agents implement the plan. Multiple subprocesses for independent tasks.
~20-120s (depends on task complexity)
Performance Note

Builders use openrouter/google/gemini-3-flash-preview by default (from lib/defaults.ts). Cheaper and faster than Opus, but still requires subprocess spawn overhead.

REVIEWloop
Reviewer audits implementation. If issues found: loop back to dispatch fixes. Max loops configurable.
~10-60s per loop iteration
Bottleneck: Review Loop

Each review iteration spawns a new subprocess AND requires the primary LLM to decide approve/reject. Worst case: multiple review loops, each adding 20-60s.

Typical Timing Breakdown (Medium Task)

UNDERSTAND
~10s
GATHER (3 scouts)
~25s
PLAN
~20s
EXECUTE (2 builders)
~35s
REVIEW
~15s

Total: ~105s for a medium complexity task through the full pipeline.

5. Subprocess Spawning

Every agent dispatch creates a new pi process. This is the fundamental mechanism for all three pipeline systems.

What happens on every spawn("pi", args)

// From agent-team.ts:dispatchAgent and agent-chain.ts:runAgent
const proc = spawn("pi", [
  "--mode", "json",       // JSON event streaming output
  "-p",                   // Pipe mode (no interactive TUI)
  "--no-extensions",      // Don't load parent's extensions
  "-e", "tasks.ts",       // Only load task discipline
  "--model", model,       // e.g. "openrouter/google/gemini-3-flash-preview"
  "--tools", tools,       // e.g. "read,grep,find,ls,bash,write,edit"
  "--thinking", "off",
  "--append-system-prompt", systemPrompt,
  "--session", sessionFile,
  task                    // The actual task text
], {
  env: { ...process.env, PI_SUBAGENT: "1" },
});

Subprocess Lifecycle

Node.js startup
~0.5s
Extension loading
~0.5-1s
Session restore
~0.3-1s
API handshake
~0.5s
Task gate ceremony
~1-2s
Actual work
variable
Overhead per subprocess: ~2-5s before any real work begins

A pipeline with 5 sequential agents pays this cost 5 times. Parallel agents pay it once (wall clock) but consume more total resources.

Task Gate Overhead

Even subagents load tasks.ts (though gating is disabled for PI_SUBAGENT=1 subprocesses for dispatch/chain tools). However, the tasks.ts extension still initializes and the agent still receives the task discipline system prompt, adding overhead to every subprocess.

6. Why It's Slow — Bottleneck Analysis

#1: Every Task Spawns Subprocesses

The primary agent is prompted to delegate. Even asking "what files are in this directory?" can trigger a subprocess spawn if the LLM decides to dispatch a scout. Each subprocess has ~2-5s startup overhead.

#2: Pipeline Has 5 Sequential Phases

When using pipeline-team, every task walks through UNDERSTAND → GATHER → PLAN → EXECUTE → REVIEW. Even simple changes go through all five phases. There's no "fast path" for trivial tasks.

#3: No Caching

Every API call is fresh. If a scout explores src/utils/ in the GATHER phase, that knowledge doesn't persist efficiently to the EXECUTE phase — it's serialized as text context ($CONTEXT) capped at 15,000 chars.

#4: Primary Agent Has No Tools (By Prompt)

While technically the primary has access to all tools, the system prompt says "You PREFER delegating codebase work to specialist agents." This means even simple reads/edits get delegated, incurring subprocess overhead.

#5: MCP Overhead (Commander)

Commander MCP adds a JSON-RPC subprocess (commander-mcp) that proxies calls over WebSocket. Every task management operation is a JSON-RPC roundtrip through this bridge: Pi → stdio → commander-mcp → WebSocket → Commander server.

#6: Task Gating Ceremony

The tasks.ts extension blocks all tool calls until the agent defines tasks via tasks new-list / tasks add. This means every interaction starts with at least one LLM turn just to define tasks before any real work begins.

Where Time Actually Goes

For a typical "add a function to utils.ts" task using the full pipeline:

LLM inference
~40%
Subprocess overhead
~25%
Task gate ceremony
~15%
API latency
~12%
File I/O
~5%
MCP roundtrips
~3%

Key insight: ~40% of total time is non-inference overhead. Subprocess spawning and task ceremony account for the majority of that overhead.

7. Simple vs Pipeline: Side-by-Side

Compare what happens for the same task: "Read utils.ts and add an isEmpty function"

Direct (agent works alone)

  1. ~0s User sends message
  2. ~2s LLM reads utils.ts directly
  3. ~3s LLM writes the function
  4. ~1s Response shown

Total: ~6s

Pipeline (full 5-phase)

  1. ~5s UNDERSTAND: LLM clarifies task, calls advance_phase
  2. ~3s Task gate: define tasks before proceeding
  3. ~15s GATHER: Spawn scout subprocess to read utils.ts
  4. ~20s PLAN: Spawn planner to create implementation plan
  5. ~25s EXECUTE: Spawn builder to write the function
  6. ~15s REVIEW: Spawn reviewer to check the code
  7. ~2s Primary summarizes for user

Total: ~85s (14x slower)

The Chain system is smarter

The agent-chain.ts system prompt explicitly tells the LLM: "Simple one-off commands: reading a file, checking status, listing contents — just do it directly." The pipeline-team has no such guidance — it always goes through all 5 phases.

8. Key Files Reference

agent/settings.json
Primary agent config: model, provider, extension loading order, theme
agent/extensions/agent-chain.ts
Sequential pipeline orchestrator. Chains agent steps with prompt templates. Provides run_chain tool.
agent/extensions/agent-team.ts
Multi-agent team dispatcher with grid dashboard. Provides dispatch_agent tool.
agent/extensions/pipeline-team.ts
5-phase hybrid pipeline (UNDERSTAND → GATHER → PLAN → EXECUTE → REVIEW). Provides advance_phase, dispatch_agents.
agent/extensions/tasks.ts
Task discipline extension. Gates all tool calls until tasks are defined. Three-state lifecycle: idle → inprogress → done.
agent/extensions/commander-mcp.ts
Bridge exposing Commander MCP tools as native Pi tools. Spawns commander-mcp subprocess, proxies JSON-RPC over stdio.
agent/extensions/toolkit-commands.ts
Registers toolkit .md files from .pi/commands/ as dynamic slash commands. Supports inline and fork execution modes.
agent/extensions/lib/mcp-client.ts
Generic JSON-RPC 2.0 client over stdio. Spawns MCP server subprocess and manages request/response lifecycle.
agent/extensions/lib/defaults.ts
Default subagent model: openrouter/google/gemini-3-flash-preview. Used when agent .md files don't specify a model.
.pi/agents/*.md
Agent definitions with frontmatter (name, description, tools, model) and system prompt body.
.pi/agents/agent-chain.yaml
Chain definitions: named sequences of agent steps with prompt templates using $INPUT / $ORIGINAL.
.pi/agents/teams.yaml
Team compositions: named groups of agents that can be activated together.
.pi/agents/pipeline-team.yaml
Pipeline configs: phase definitions with mode (interactive/parallel/sequential) and agent assignments.
