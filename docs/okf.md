# Open Knowledge Format (OKF) Reference

> A guide to working with OKF knowledge bases in Arcanum and Guild-enabled projects.

## What is OKF?

**Open Knowledge Format v0.1** is a specification for agent-readable knowledge bases. It is:

- A directory of markdown files with YAML frontmatter
- Authored by humans, generated by agents, diffable in git, and portable across tools
- Designed to be queried by agents instead of blindly reading large numbers of files
- Inspired by [Andrej Karpathy's LLM wiki pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

**Official spec:** [GoogleCloudPlatform/knowledge-catalog](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)

## Why Arcanum Uses OKF

### Token Efficiency
Heavy lifting — understanding relationships, synthesizing patterns, discovering conventions — is done once and reused. Agents query specific paths instead of reading multiple raw files.

### Knowledge Compounds Over Time
Every discovery (architectural decision, convention, pitfall) is captured in the knowledge base. The bundle grows richer and more valuable as the project evolves.

### Versionable and Portable
The knowledge base lives in git alongside the code. It can be migrated between projects, shared across teams, and reviewed in pull requests.

### Human and Agent Readable
Both humans and agents can navigate, contribute to, and benefit from the same knowledge base.

## How OKF Works in Guild

When you run `guild-init` in a project, it scaffolds an OKF bundle at `.guild/knowledge/`:

```
.guild/knowledge/
├── index.md        # Bundle root — agents start here
├── decisions.md    # Architectural decisions log
├── conventions.md  # Coding standards and patterns
└── gotchas.md      # Known pitfalls and fix patterns
```

### Initialization Behavior

- `guild-init` creates the bundle structure on first run
- On subsequent runs, existing files are **never overwritten** — they accumulate knowledge
- New files can be added to the bundle at any time

### How Agents Use It

1. Agents load `index.md` first to understand the bundle structure
2. They navigate to specific files based on the index
3. They update relevant files when discovering new conventions, decisions, or pitfalls
4. They never delete entries — they mark them superseded or deprecated instead

## OKF Document Format

Every markdown file in the bundle (except `index.md` and `log.md`) must have YAML frontmatter:

```yaml
---
type: <concept type>        # REQUIRED — e.g. "Reference", "Decision Log", "Playbook"
title: <display name>       # Recommended
description: <one-liner>    # Recommended
timestamp: <ISO 8601>       # Recommended
---
```

### Example

```yaml
---
type: Decision Log
title: Architectural Decisions
description: Record of significant architectural choices and their rationale
timestamp: 2024-01-15T10:30:00Z
---

## ADR-001: Use TypeScript for type safety

**Date:** 2024-01-15  
**Status:** Accepted

We chose TypeScript to catch type errors at compile time and improve IDE support.
```

### Reserved Filenames

| Filename | Purpose | Frontmatter Required |
|----------|---------|---------------------|
| `index.md` | Bundle root; agents start here | No (optional `okf_version`) |
| `log.md` | Chronological update history | No |
| All others | Domain-specific knowledge | Yes |

## How to Contribute

### Adding a New Architectural Decision

Edit `.guild/knowledge/decisions.md`:

```markdown
## ADR-NNN: <Title>

**Date:** YYYY-MM-DD  
**Status:** Proposed / Accepted / Superseded / Deprecated

<Rationale and context>

**Consequences:**
- <Impact 1>
- <Impact 2>
```

### Adding a New Convention

Edit `.guild/knowledge/conventions.md`:

```markdown
## <Convention Name>

<Description of the convention>

**Example:**
```typescript
// Good
const getUserById = (id: string) => { ... }

// Bad
const get_user_by_id = (id: string) => { ... }
```
```

### Adding a Known Pitfall

Edit `.guild/knowledge/gotchas.md`:

```markdown
## <Pitfall Name>

**Problem:** <What goes wrong>

**Root cause:** <Why it happens>

**Fix:** <How to resolve it>

**Prevention:** <How to avoid it in the future>
```

### Adding a New Concept File

1. Create a new `.md` file in `.guild/knowledge/`
2. Add YAML frontmatter with `type`, `title`, and `description`
3. Link it from `index.md` under an appropriate section

## Best Practices

### Keep Entries Atomic
Each entry should address one concept. Link to related entries instead of duplicating content.

### Never Delete — Mark Superseded
When a decision or convention changes, mark the old entry as superseded:

```markdown
## <Old Convention>

**Status:** Superseded by <New Convention> (see ADR-NNN)

<Original content>
```

### Use Clear Timestamps
Always include ISO 8601 timestamps (`YYYY-MM-DDTHH:MM:SSZ`) so agents can understand recency.

### Link Between Files
Use relative markdown links to connect related concepts:

```markdown
See also: [Naming Conventions](./conventions.md#naming)
```

### Keep Descriptions Concise
Frontmatter descriptions should be one-liners. Save detailed explanations for the body.

## Integration with Agent Instructions

Add this to your project's `AGENTS.md` to teach agents how to use the knowledge base:

```markdown
## Knowledge Base

For any question about the codebase, first navigate the knowledge base at `.guild/knowledge/index.md`.
Use specific paths from the index instead of reading multiple raw files.
Update the relevant knowledge file whenever you discover a new convention, decision, or pitfall.
```

## Complementary Tools

OKF works well alongside other context tools:

| Tool | Purpose | Complements OKF |
|------|---------|-----------------|
| **Graphify** | Structural code graph | OKF captures *why*; Graphify shows *what* |
| **AGENTS.md** | Behavioral guidelines | OKF captures project-specific knowledge; AGENTS.md teaches agent behavior |
| **Codebase comments** | Inline documentation | OKF captures patterns; comments explain specific code |

## Troubleshooting

### Agents Ignore the Knowledge Base
- Verify `AGENTS.md` includes the knowledge base instruction (see Integration section above)
- Ensure `index.md` exists and is readable
- Check that frontmatter is valid YAML

### Files Keep Getting Overwritten
- `guild-init` should never overwrite existing files. If it does, report a bug.
- Verify `.guild/knowledge/` is not in `.gitignore` (it should not be)

### Knowledge Base Grows Too Large
- Split large files (e.g., `conventions.md`) into multiple focused files
- Archive old decisions to a separate `decisions-archive.md`
- Link from `index.md` to all files so agents can navigate efficiently
